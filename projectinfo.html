01 — Overview

System Overview

Where is My Bus is a crowdsourced real-time bus tracking platform for Kerala KSRTC. Unlike GPS trackers installed on buses, this system works by passively collecting GPS data from passengers' phones and aggregating it into a live bus location feed — similar to how "Where is My Train" works for Indian Railways.

💡
Core idea: Every user with the app installed becomes a passive GPS sensor. The server aggregates thousands of GPS pings to compute real-time bus positions — no hardware installation on buses required.

03 — Core GPS Flow

How Crowdsourced GPS Works

This is the heart of the system. Every user's phone becomes a passive sensor that contributes to the live bus map.

1
User Boards Bus
App detects motion pattern + GPS near route
2
GPS Logged
Phone silently logs lat/lng every 15s in background
3
Batch Upload
Queued pings sent when network available
4
Map Matching
Server snaps GPS to nearest bus route
5
Aggregation
Multiple passengers' pings merged into one location
6
Live Push
Bus location broadcast to all viewers via WebSocket
// GPS Processing Algorithm (simplified) function processPing(ping) { // 1. Validate speed (buses: 0-90 km/h) if (ping.speed > 90 || ping.speed < 0) return 'invalid'; // 2. Snap to nearest KSRTC route polyline const matched = mapMatch(ping.lat, ping.lng, ksrtcRoutes); // 3. Identify which bus (cluster nearby pings) const busId = clusterToBus(matched, activeBuses); // 4. Kalman filter to smooth noisy GPS const smoothed = kalmanFilter(busId, matched); // 5. Update Redis with latest position redis.set(`bus:${busId}:location`, smoothed, 'EX', 30); // 6. Broadcast to WebSocket subscribers io.to(`route:${matched.routeId}`).emit('bus:update', smoothed); }

05 — API Design

Core API Endpoints

GET /api/buses/nearby?lat=&lng=&radius= Get buses within radius of user location
GET /api/buses/:busId/location Get current GPS location of a specific bus
GET /api/routes/:routeId/stops Get all stops for a route with ETA
POST /api/gps/ping Submit GPS coordinates from passenger's phone
POST /api/gps/batch Bulk upload queued pings (offline sync)
GET /api/stops/:stopId/arrivals Get upcoming bus arrivals at a stop
WS ws://api/live/route/:routeId WebSocket stream of live bus positions on route
WS ws://api/live/stop/:stopId WebSocket for ETA updates at a stop

07 — Android GPS Strategy

Android GPS Architecture

The Android app uses Google's Fused Location Provider API which intelligently combines GPS, WiFi, and cell tower data to get the most accurate location while minimising battery drain.

// Android: Background GPS logging with WorkManager class GpsWorker(context: Context, params: WorkerParameters) : CoroutineWorker(context, params) { override suspend fun doWork(): Result { val fusedClient = LocationServices.getFusedLocationProviderClient(applicationContext) // High accuracy: GPS + network hybrid val request = LocationRequest.create() .setPriority(PRIORITY_HIGH_ACCURACY) .setInterval(15_000) // Every 15 seconds .setSmallestDisplacement(20f) // Only if moved 20m // Store locally if offline, upload when connected val ping = GpsPing(lat, lng, speed, accuracy, System.currentTimeMillis()) localDb.insert(ping) if (isNetworkAvailable()) api.uploadBatch(localDb.getPending()) return Result.success() } }
⚠️
Battery optimization: Use setSmallestDisplacement(20f) to skip pings when the bus hasn't moved. This reduces battery usage by ~60% on slow/stuck buses.
📡
Network Location fallback: In tunnels or poor GPS areas, the Fused API automatically falls back to cell tower triangulation — still accurate to ~50m, which is enough for bus tracking.
